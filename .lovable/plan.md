
## Что делаем

1. **SSG-сборка** — статические HTML для всех маршрутов + `handler.php` рядом, чтобы залить на обычный shared-хостинг (Apache/nginx + PHP).
2. **Единая форма** — один компонент `ContactForm`, который вызывается на нужных страницах. Меняешь в одном месте — меняется везде.
3. **Отправка** — форма шлёт POST на `/handler.php` (PHP отправляет письмо через PHPMailer и сообщение в Telegram).
4. **Правовые страницы** — 3 новых маршрута с рыбой-текстом + чекбокс согласия в форме.

---

## 1. Перевод проекта в SSG

В `vite.config.ts` включаем prerender всех маршрутов и отключаем серверный рантайм (target: static):

```ts
export default defineConfig({
  tanstackStart: {
    target: 'static',
    prerender: { enabled: true, crawlLinks: true, routes: ['/'] },
  },
});
```

После `bun run build` в `dist/` (или `.output/public/`) получаем готовый набор `.html` файлов: `index.html`, `product/index.html`, `applications/index.html` и т.д. — заливаются прямо в `public_html` хостинга.

Уберём из проекта то, что не работает в SSG-режиме:
- `src/server.ts` (Cloudflare Worker wrapper) — не нужен для статики.
- В `__root.tsx` оставляем только статический shell, без серверных вызовов.

SEO (`head()` с title/description/og) уже на каждой странице — отлично переживёт prerender.

## 2. handler.php генерируется при каждом билде

Положу шаблон `handler.template.php` в `scripts/`, а в `package.json` добавлю postbuild-скрипт:

```
"build": "vite build && node scripts/emit-php.mjs"
```

Скрипт `scripts/emit-php.mjs`:
- копирует `handler.template.php` → `dist/handler.php`;
- (опционально) кладёт пример `config.example.php` в `dist/` для удобства первой настройки на хостинге.

В `handler.php` будет твой код + добавлю отправку в Telegram (используя `telegram_token` и `telegram_chat_id` из `config.php`) — одним cURL-запросом к `api.telegram.org/bot<token>/sendMessage` после успешной отправки письма. Если письмо упало, заявка всё равно уйдёт в Telegram (и наоборот) — пользователь получит `success: true`, если сработал хотя бы один канал.

`config.php` лежит уровнем выше `public_html` — в репо его не кладём, в `dist` тоже. Дам тебе готовый `config.example.php` в README с инструкцией.

## 3. Единый компонент формы

`src/components/ContactForm.tsx` уже существует. Доработки:
- Добавить поле `details` (textarea, необязательное) — оно ожидается в PHP-обработчике.
- Добавить чекбокс «Я соглашаюсь с [пользовательским соглашением](/terms) и [обработкой персональных данных](/privacy)» — обязательный, валидируется zod.
- Заменить toast-заглушку на реальный `fetch('/handler.php', { method: 'POST', body: JSON.stringify({name, phone, details}) })`.
- Принимать пропсы: `title?`, `description?`, `compact?` — чтобы один компонент красиво смотрелся и в карточке на `/contacts`, и в CTA-секции на других страницах.

Везде, где сейчас «оставьте заявку», вызываем именно этот компонент. Единственный источник правды.

Куда подключаем форму сейчас (помимо `/contacts`):
- `/` (главная) — в CTA-секции внизу.
- `/product`, `/delivery`, `/installation` — CTA «Запросить расчёт» в конце страницы.

## 4. Три правовые страницы (рыба-текст)

Новые маршруты:
- `src/routes/terms.tsx` — «Пользовательское соглашение»
- `src/routes/privacy.tsx` — «Политика обработки персональных данных»
- `src/routes/cookies.tsx` — «Политика cookie»

Каждая страница — простой layout: заголовок + lorem-блоки с разделами (Общие положения / Термины / Права и обязанности / Ответственность / Контакты), дата актуализации. Свой `head()` с title/description.

В `Footer.tsx` добавлю блок ссылок «Документы» со ссылками на все три страницы. В навигацию верхнего меню не добавляем.

## 5. Чекбокс согласия — поведение

- По умолчанию выключен. Кнопка «Отправить» дизейблится, пока чекбокс не отмечен.
- Текст: «Я соглашаюсь с [пользовательским соглашением] и [обработкой персональных данных]» — оба слова-ссылки открывают соответствующие страницы в новой вкладке.
- При сабмите zod-схема требует `consent === true`, иначе ошибка под чекбоксом.

## Технические детали

- **Сборка SSG.** TanStack Start prerender обходит все маршруты и пишет HTML; маршруты с `loader`-ом для prerender должны работать без серверной авторизации (у нас все публичные — ок).
- **CORS не нужен.** `handler.php` лежит на том же домене, что и сайт.
- **Безопасность.** В PHP-обработчике добавлю минимальную валидацию (длина имени/телефона, очистка `details` от слишком длинных значений) и `htmlspecialchars` уже у тебя есть.
- **Telegram.** Сообщение формата:
  ```
  🔔 Новая заявка
  Имя: {name}
  Телефон: {phone}
  Детали: {details}
  ```
- **Куда заливать на хостинг.** Содержимое `dist/` → в `public_html/`. Файл `config.php` → на один уровень выше `public_html/`. Папку `PHPMailer/` (скачивается отдельно с GitHub релиза) → тоже на уровень выше `public_html/`. Дам короткий `DEPLOY.md` с инструкцией.

## Чего НЕ делаем сейчас

- Lovable Cloud / серверные функции — не нужны, всё на PHP.
- Аутентификацию / админку заявок — заявки уходят на почту и в Telegram.
- Реальный текст юр.документов — только рыба. Финальный текст подложишь сам.
